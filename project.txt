ABSTRACT 
Hospitals continue to face increasing challenges in maintaining patient safety and 
ensuring efficient supervision within wards. Traditional monitoring methods rely heavily 
on manual oversight by clinical staff, making it difficult to detect high-risk interactions, 
prevent unauthorized movement, monitor contagious patients, or respond quickly to 
emergencies. Incidents reported in major hospitals, such as patients harming one 
another, absconding without settlement of bills, or accessing restricted areas, highlight 
the urgent need for a more automated and reliable monitoring solution. 
This project focuses on the General Medical Ward at Aga Khan University Hospital as a 
model environment. This ward presents real-world conditions such as patients with 
contagious diseases, patients with unpredictable or violent behavior, and individuals 
who may wander or leave without authorization. The proposed system utilizes wearable 
sensor-based wristbands that transmit patient identity and movement data in real time. 
These wristbands alert the staff to possible removal attempts if tampered with. 
Receiver nodes installed at strategic locations corridors, entry/exit points, and 
restricted zones detect patient movements and transmit the information to a 
centralized database. Medical staff will access a dashboard or mobile application to 
view patient locations, movement history, alerts, and zone violations. This system aims 
to enhance operational safety, reduce the workload on health workers, improve 
emergency response time, and provide a centralized record of patient movement for 
auditing and analysis. 
1. Objective 
1.1 Main Objective 
To design and develop an automated patient tracking and safety monitoring system for 
hospital wards using wearable tags and a centralized monitoring platform. 
1.2 Specific Objectives 
a) To develop a wearable patient wristband equipped with a sensor for transmitting 
identity and movement data. 
b) To install and configure receiver nodes for tracking patient movement across different 
hospital ward zones. 
c) To design a secure backend database for logging movements, alerts, and patient 
zone-history. 
d) To develop a user interface (dashboard/mobile view) for healthcare workers to 
monitor patient location in real time. 
e) To implement automated alerts for restricted-area access, removal of wristbands, 
risky interactions, and unauthorized exits.

Hospital General Medical Ward.

• Testing system performance and validating movement-logging data. 
• Preparing system documentation and final presentation. 
3. Learning Areas 
a) Wireless communication technologies such as RFID, BLE, or UWB. 
b) Real-time tracking and localization methods in indoor environments. 
c) Database design and implementation for live logging systems. 
d) User interface and dashboard development for monitoring applications. 
e) Safety protocols and movement-control strategies in hospital wards. 
f) System integration and testing in a model environment. 
4. Benefits of the Project if Successful 
a) Enhances patient safety by reducing incidents of violence, unauthorized movement, 
and absconding. 
b) Improves monitoring of contagious patients and minimizes exposure risks. 
c) Provides healthcare staff with real-time visibility, reducing manual supervision 
workload. 
d) Strengthens emergency response by tracking patient location instantly. 
e) Offers long-term records for auditing, research, and administrative decision-making. 
f) Serves as a scalable model that can be replicated across different hospital wards or 
facilities. 
5. Methodology 
• Conducting a study of movement zones within the General Medical Ward (entry, exit, 
rooms, corridors, restricted areas). 
• Designing a scaled-down physical model of the ward layout.

• Simulating patient activities within the model to test detection accuracy and event 
logging.

## 2. HARDWARE SPECIFICATIONS

### 2.1 WEARABLE WRISTBAND UNIT

#### 2.1.1 Main Controller
**Component:** ESP32-WROOM-32 or ESP32-PICO-D4  
**Specifications:**
- Dual-core Xtensa LX6 @ 240MHz
- 520KB SRAM, 4MB Flash
- WiFi 802.11 b/g/n (2.4GHz)
- Bluetooth v4.2 BR/EDR and BLE
- Operating Voltage: 3.3V
- Deep Sleep Current: ~10μA
- Active Current: ~160mA (WiFi TX)

**Why ESP32:**
- Built-in WiFi for firmware OTA updates
- Multiple I2C interfaces for sensors
- Low power modes for battery optimization
- Sufficient processing power for edge computing

**Selected:** 125kHz Passive RFID Tag (integrated into wristband)
- Unique ID: 8-10 digit hexadecimal
- Programmable patient identification

#### 2.1.3 Motion & Orientation Sensor
**Component:** MPU6050 (6-Axis IMU)  
**Specifications:**
- 3-axis Gyroscope: ±250, ±500, ±1000, ±2000°/s
- 3-axis Accelerometer: ±2g, ±4g, ±8g, ±16g
- Interface: I2C (400kHz Fast Mode)
- Operating Voltage: 3.3V
- Operating Current: 3.9mA
- I2C Address: 0x68 (default) or 0x69

**Pin Connections:**
- VCC → 3.3V
- GND → GND
- SDA → GPIO21
- SCL → GPIO22
- INT → GPIO19 (interrupt for motion detection)

**Detection Capabilities:**
- Fall detection (sudden acceleration change)
- Wristband removal (orientation change)
- Abnormal movement patterns
- Patient activity level monitoring

#### 2.1.4 Vital Signs Sensor
**Component:** MAX30102 (Heart Rate + SpO2 Sensor)  
**Specifications:**
- Measurement: Heart rate (BPM) and Blood Oxygen (SpO2%)
- Interface: I2C
- Operating Voltage: 3.3V
- Supply Current: 600µA @ 25Hz sampling
- I2C Address: 0x57

**Pin Connections:**
- VIN → 3.3V
- GND → GND
- SDA → GPIO21 (shared I2C bus with MPU6050)
- SCL → GPIO22 (shared I2C bus with MPU6050)
- INT → GPIO18 (interrupt when new data ready)

**Monitoring Features:**
- Continuous heart rate monitoring (50-100 BPM normal range)
- Blood oxygen saturation (95-100% normal)
- Alerts for abnormal readings
- Confirms wristband is being worn

#### 2.1.5 Temperature Sensor
**Component:** DHT11 Temperature & Humidity Sensor  
**Specifications:**
- Temperature Range: 0-50°C (±2°C accuracy)
- Humidity Range: 20-90% RH (±5% accuracy)
- Interface: Single-wire digital
- Operating Voltage: 3.3V - 5V
- Sampling Rate: 1Hz (once per second)

**Pin Connections:**
- VCC → 3.3V
- GND → GND
- DATA → GPIO4
- Pull-up resistor: 10kΩ between DATA and VCC

**Monitoring Purpose:**
- Body temperature monitoring for fever detection
- Environmental temperature for contagious patient isolation
- Alert threshold: >37.5°C (fever indication)

#### 2.1.6 Tamper Detection Sensors

**A. Conductive Thread Strap Break Detection**  
**Implementation:**
- Conductive thread (silver-coated nylon) sewn into wristband strap
- Creates continuous circuit when strap is intact
- Connected to GPIO15 with pull-up resistor (10kΩ)
- Circuit breaks when strap is cut or torn

**Pin Connection:**
- Thread Start → 3.3V
- Thread End → GPIO15 (INPUT_PULLUP)
- Detection: LOW = intact, HIGH = broken

**B. Skin Contact Sensor**  
**Component:** Capacitive Touch Sensor (ESP32 Built-in)  
**Implementation:**
- Copper pad on inner wristband surface
- Connected to ESP32 touch-sensitive GPIO
- Detects skin conductivity

**Pin Connection:**
- Touch Pad → GPIO15 (T3 - Touch Pin 3)
- No external components needed

**Detection Logic:**
- Continuous touch reading (0-100 scale)
- Skin contact: Reading < 40
- No contact: Reading > 60
- Alert triggered when no contact for >10 seconds

#### 2.1.7 User Feedback Indicators

**LED Indicator**
- Component: 5mm Green/Red Dual-color LED
- Pin: GPIO2
- Current-limiting resistor: 220Ω
- Status Codes:
  - Solid Green: Normal operation
  - Blinking Green: Transmitting data
  - Solid Red: Low battery
  - Blinking Red: Alert/tamper detected
  - Off: Deep sleep mode

**Buzzer (Optional)**
- Component: Passive buzzer (5V tolerant, 3.3V operated)
- Pin: GPIO5
- Purpose: Local alert for patient (medication time, etc.)
- Frequency: 2kHz
- Usage: Minimal to save battery

#### 2.1.8 Power System

**Battery:**
- Type: 3.7V Li-Po Rechargeable
- Capacity: 500mAh - 1000mAh
- Dimensions: 40mm × 30mm × 5mm (approx)
- Protection: Over-charge, over-discharge, short-circuit

**Charging Module:**
- Component: TP4056 Li-Po Charging Board
- Input: 5V Micro-USB
- Charging Current: 500mA (adjustable)
- LED Indicators: Red (charging), Blue (charged)

**Power Distribution:**
- Battery → TP4056 OUT+ → ESP32 VIN (or 3.3V via LDO)
- Battery → TP4056 OUT- → Common GND
- All sensors powered from ESP32 3.3V rail

**Estimated Battery Life:**
- Active Mode (WiFi + Sensors): 8-12 hours
- RFID Mode (ESP32 deep sleep): 3-5 days
- Hybrid Mode (RFID + periodic WiFi): 1-2 days

#### 2.1.9 Wristband Enclosure
**Material:** Medical-grade silicone or ABS plastic  
**Requirements:**
- Waterproof rating: IP65 minimum
- Hypoallergenic material
- Adjustable strap (fits 15-22cm wrist circumference)
- Secure clasp with conductive thread integration
- Access port for USB charging
- Dimensions: ~50mm × 35mm × 15mm

#### 2.1.10 Wristband PCB Layout
**Form Factor:** Custom PCB (40mm × 30mm)  
**Layers:** 2-layer PCB  
**Components Placement:**
- Top: ESP32, MAX30102, MPU6050, connectors
- Bottom: RFID tag antenna, battery connector, TP4056

├── GPIO21 (SDA) ───────► MPU6050 SDA, MAX30102 SDA

## 3. SOFTWARE ARCHITECTURE

### 3.1 WRISTBAND FIRMWARE (ESP32)

#### 3.1.1 Operating Modes

**Mode 1: RFID-Only Mode (Default)**
- ESP32 in deep sleep (10μA)
- RFID tag continuously broadcasting ID
- Wake up every 30 minutes for sensor check
- Battery life: 3-5 days

**Mode 2: Active Monitoring Mode**
- ESP32 active, all sensors reading
- RFID broadcasting
- WiFi enabled for direct data transmission
- Battery life: 8-12 hours
- Use case: High-risk patients, isolation wards

**Mode 3: Hybrid Mode**
- RFID broadcasting continuously
- ESP32 wakes every 5 minutes for sensor check
- WiFi enabled only for alerts
- Battery life: 1-2 days

**Mode 4: Charging Mode**
- All non-critical functions suspended
- LED indicates charging status
- Firmware update capability via USB

#### 3.1.2 Firmware Architecture

```
Main Loop Flow:
├── Initialize Hardware
│   ├── ESP32 initialization
│   ├── I2C bus setup (MPU6050, MAX30102)
│   ├── GPIO configuration (DHT11, sensors, LED)
│   └── RFID tag activation
│
├── Sensor Reading Loop (Every 5 seconds)
│   ├── Read MPU6050 (acceleration, orientation)
│   ├── Read MAX30102 (heart rate, SpO2)
│   ├── Read DHT11 (temperature)
│   ├── Check strap integrity
│   ├── Check skin contact
│   └── Process sensor data
│
├── Alert Detection Logic
│   ├── Fall detection (sudden acceleration change)
│   ├── Tamper detection (strap break or skin contact loss)
│   ├── Abnormal vitals (HR, SpO2, temp out of range)
│   └── Set alert flags
│
├── Data Transmission (Mode-dependent)
│   ├── RFID: Continuous broadcast
│   ├── WiFi: Send data via HTTPS (if alert or scheduled)
│   └── LED/Buzzer feedback
│
└── Power Management
    ├── Battery level monitoring
    ├── Sleep mode activation
    └── Low battery alert
```

#### 3.2.3 Zone Detection Logic

Each receiver node is configured with:
- Zone name (e.g., "Corridor-1", "Exit Door", "Medication Room")
- Zone type (e.g., "normal", "restricted", "exit")
- Alert rules (e.g., "alert if unauthorized patient detected")

```cpp
// Example zone configuration
struct Zone {
  String zone_name;
  String zone_type; // "normal", "restricted", "exit", "isolation"
  bool require_authorization;
  int max_occupancy;
  String[] authorized_patients;
};

// Check if detection should trigger alert
bool checkZoneViolation(String patient_id, Zone zone) {
  if (zone.zone_type == "restricted" && 
      !isAuthorized(patient_id, zone.authorized_patients)) {
    return true; // Trigger alert
  }
  return false;
}
```
